## 一、Redis 接入（缓存与限流基础）

**目标**：为后续接口限流和业务缓存提供 Redis 基础设施。

### 变更清单

| 文件 | 动作 | 说明 |
| --- | --- | --- |
| `pom.xml` | 修改 | 新增 `spring-boot-starter-data-redis` + `commons-pool2` |
| `application.properties` | 修改 | 新增 Redis 连接 + Lettuce 连接池配置 |
| `config/RedisConfig.java` | **新建** | RedisTemplate 和 StringRedisTemplate Bean 配置 |

### 技术选型

- **客户端**：Lettuce（Spring Boot 3 默认，Netty 异步、线程安全）
- **序列化方案**：
  - Key → `StringRedisSerializer`（避免 Redis 中的 `\xac\xed\x00\x05` 乱码）
  - Value → `GenericJackson2JsonRedisSerializer`（JSON 格式，注册 `JavaTimeModule` 支持 `LocalDateTime`）
- **未启用 `defaultTyping`**：不注入 `@class` 字段，数据更干净
- **ObjectMapper 独立构造**：不污染前端 Jackson 配置（Long → String 等）

### 连接池配置

```properties
spring.data.redis.host=localhost
spring.data.redis.port=6379
spring.data.redis.lettuce.pool.max-active=16
spring.data.redis.lettuce.pool.max-idle=8
spring.data.redis.lettuce.pool.min-idle=2
```

## 二、接口限流（防刷）

**目标**：基于 Redis 实现接口级别的滑动窗口限流，防止恶意刷接口。

### 变更清单

| 文件 | 动作 | 说明 |
| --- | --- | --- |
| `pom.xml` | 修改 | 新增 `spring-boot-starter-aop` 依赖 |
| `common/result/ResultCode.java` | 修改 | 新增 `RATE_LIMITED(429, "请求过于频繁，请稍后再试")` |
| `common/annotation/RateLimit.java` | **新建** | `@RateLimit` 注解（参数：time、maxCount、message） |
| `common/aspect/RateLimitAspect.java` | **新建** | AOP 切面，Lua 脚本 + Redis ZSET 滑动窗口 |

### 核心设计

```
请求 → @Around("@annotation(rateLimit)")
        → 构建 Key: rate_limit:{类名}:{方法名}:{userId 或 IP}
        → Lua 原子脚本: ZREMRANGEBYSCORE → ZCARD → ZADD / 拒绝
        → 通过: proceed()
        → 拒绝: throw BusinessException(429)
        → GlobalExceptionHandler 统一返回 {"code":429,...}
```

### 技术细节

- **算法**：ZSET 滑动窗口（非固定窗口），消除边界突增问题
- **原子性**：Lua 脚本在 Redis 服务端原子执行，避免并发竞态
- **用户标识**：已登录按 `userId` 限流，未登录按 IP 限流（X-Forwarded-For / X-Real-IP / RemoteAddr）
- **Key 隔离**：不同 Controller、不同方法、不同用户/IP 的限流互相独立
- **Key 自动过期**：窗口时间结束后 Redis Key 自动清除，不积累垃圾数据

### 使用示例

```java
@RateLimit(time = 60, maxCount = 3, message = "发帖太频繁，请稍后再试")
@PostMapping
public Result<Long> createPost(@Validated @RequestBody PostDTO postDTO) { ... }
```

### 测试验证

```bash
curl localhost:8081/api/boards  # 1 → 200
curl localhost:8081/api/boards  # 2 → 200
curl localhost:8081/api/boards  # 3 → 429 "测试限流"
```

## 三、XSS 安全防护

**目标**：对前端传入的所有 JSON String 字段自动清洗 XSS 攻击载荷，同时为富文本保留安全 HTML 标签。

### 变更清单

| 文件 | 动作 | 说明 |
| --- | --- | --- |
| `pom.xml` | 修改 | 新增 `jsoup 1.18.1` 依赖 |
| `common/annotation/SkipXssClean.java` | **新建** | 标注字段跳过 XSS 清洗（如 password） |
| `common/handler/XssStringDeserializer.java` | **新建** | Jackson 全局 String 反序列化器 |
| `config/JacksonConfig.java` | 修改 | 注册 XssCleanModule Bean |
| `module/auth/entity/RegisterDTO.java` | 修改 | `password` 字段加 `@SkipXssClean` |
| `module/auth/entity/LoginDTO.java` | 修改 | `password`、`account` 字段加 `@SkipXssClean` |

### 核心设计

```
请求 JSON → Jackson 反序列化
           → 遇到 String 类型 → XssStringDeserializer
           → 检测 @SkipXssClean 注解
              ├─ 有注解 → 跳过清洗（适用于 password）
              └─ 无注解 → Jsoup.clean(value, Safelist.relaxed())
           → Controller 拿到已清洗的 DTO
```

### 技术选型对比

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| **Jackson 反序列化器** ✅ | 零性能开销、天然支持嵌套对象、字段级精确跳过 | 不覆盖 @RequestParam |
| Filter + HttpServletRequestWrapper | 覆盖所有输入 | 需完整读→解析→序列化 body，性能差；复杂 JSON 易漏 |

**选择 Jackson 方案**：项目 95%+ 的 String 输入走 `@RequestBody`，且不与现有 Sa-Token 拦截器、Jackson 配置产生冲突。

### 清洗规则

- **移除**：`<script>`、`onerror`、`onclick`、`<iframe>`、`<object>`、`<embed>` 等危险标签和事件属性
- **保留**：`<b>`、`<i>`、`<u>`、`<p>`、`<br>`、`<a href>`、`<img src>` 等安全富文本标签（`Safelist.relaxed()`）
- **跳过字段**：`password`、`account`（标注 `@SkipXssClean`，避免修改认证凭据）

### 测试验证

| 输入 | 输出 | 结论 |
| --- | --- | --- |
| `<script>alert(1)</script><p>文本</p>` | `<p>文本</p>` | 危险标签清除，安全标签保留 |
| `<b>加粗</b><a href="...">链接</a>` | 原样保留 | 安全标签不误杀 |
| `<img src=x onerror=alert(1)>` | `<img>` | 事件属性清除 |
| `<script>hack</script>pass123`（password） | 原样保留 | @SkipXssClean 生效 |

### 与后续管道的关系

```
Jackson XSS 清洗 → @Valid 参数校验 → SensitiveWordUtil 敏感词过滤 → Service → DB
```

## 四、敏感词过滤

**目标**：在内容入库前自动检测和替换敏感词，净化社区内容。

### 变更清单

| 文件 | 动作 | 说明 |
| --- | --- | --- |
| `common/util/SensitiveWordUtil.java` | **新建** | DFA（Trie 树）算法 + 内置 25 个敏感词 + FilterResult |
| `post/service/impl/PostServiceImpl.java` | 修改 | title + content 入库前过滤，命中打 warn 日志 |
| `interact/service/impl/CommentServiceImpl.java` | 修改 | content 入库前过滤，命中打 warn 日志 |
| `message/service/impl/MessageServiceImpl.java` | 修改 | content 入库前过滤，命中打 warn 日志 |

### 核心设计

```java
SensitiveWordUtil.filter(text) → FilterResult {
    filteredText:  "这是一条***广告***信息",
    hasSensitive:  true,
    matched:       ["赌博", "博彩"]
}
```

### 技术细节

- **算法**：DFA 有限状态自动机（Trie 树），时间复杂度 O(n)，与词库大小无关
- **匹配策略**：最长匹配原则（`博彩` 优先于 `博`）
- **替换方式**：统一替换为 `***`（固定长度，不暴露原词长度），不做直接删除（避免语句不通）
- **日志记录**：命中时打印 WARN 级别日志，记录 userId 和命中的词列表，便于运维审计

### 内置词库（25 个，分 6 类）

| 类别 | 示例 |
| --- | --- |
| 色情低俗 | 色情、淫秽、裸聊、成人视频、约炮、黄片、av女优 |
| 赌博 | 赌博、博彩、赌场、六合彩、押注、时时彩 |
| 广告引流 | 兼职日结、加微信、扫码领、点击领取、免费红包 |
| 政治敏感 | 法轮功、台独、藏独、疆独、falungong |
| 辱骂攻击 | 傻逼、他妈的、操你妈、去死吧 |
| 违法 | 代办证件、枪支弹药、毒品 |

> 后续可从数据库加载词库，调用 `SensitiveWordUtil.addWords()` 动态扩展。

## 五、图形验证码

**目标**：登录接口增加图形验证码校验，防止自动化撞库和爆破攻击。

### 变更清单

| 文件 | 动作 | 说明 |
| --- | --- | --- |
| `pom.xml` | 修改 | 新增 `hutool-captcha 5.8.36` 依赖 |
| `module/auth/entity/LoginDTO.java` | 修改 | 新增 `captchaCode`、`captchaUuid` 字段（`@NotBlank`） |
| `module/auth/controller/AuthController.java` | 修改 | 新增 `GET /api/auth/captcha`；login 方法增加验证码校验 |
| `config/SaTokenConfig.java` | 修改 | `/api/auth/captcha` 加入白名单 |

### 核心设计

```
用户 → GET /api/auth/captcha
      → Hutool ShearCaptcha 生成扭曲验证码（150×50，4 字符）
      → 验证码原文存入 Redis key: captcha:{uuid}，TTL 2 分钟
      → 返回 { uuid, base64 } → 前端渲染 <img>

用户 → POST /api/auth/login { account, password, captchaCode, captchaUuid }
      → 从 Redis 取出 captcha:{captchaUuid}
      → 不区分大小写比对 → 校验通过立即 DELETE Redis key（一次性消费）
      → 校验失败 → BusinessException(400, "验证码错误或已过期")
```

### 技术细节

- **验证码生成**：Hutool `ShearCaptcha`（旋转+扭曲干扰），字符池 0-9a-zA-Z
- **存储**：Redis String，Key 格式 `captcha:{uuid}`，2 分钟 TTL
- **一次性消费**：验证通过后立即删除 Redis key，防止验证码复用
- **校验顺序**：验证码校验在密码校验之前执行，阻断无效密码的数据库查询
- **Base64 图片**：前端可直接 `<img src="data:image/png;base64,...">` 无需额外请求

### WebSocket 群聊协议

```
连接地址: ws://localhost:8081/ws/chat/{groupId}?satoken={token}

握手校验:
  1. HandshakeInterceptor 从 query 参数提取 satoken
  2. StpUtil.getLoginIdByToken(token) 校验有效性
  3. 校验用户是否为该群成员（非成员拒绝握手）
  4. 通过后 userId 存入 session.attributes

消息格式:
  客户端 → 服务端: {"content": "大家好！"}
  服务端 → 客户端（type=message）: {"type":"message","data":{...完整消息JSON...}}
  服务端 → 客户端（type=system）: {"type":"system","data":{"content":"...","onlineCount":"3"}}
  服务端 → 客户端（type=error）:  {"type":"error","data":{"content":"错误描述"}}
```

### 核心架构

```
                          ChatWebSocketHandler
                               │
              ConcurrentHashMap<Long, Set<WebSocketSession>>
                        (groupId → 在线会话集合)
                               │
        ┌──────────────────────┼──────────────────────┐
        ▼                      ▼                      ▼
   连接建立                消息处理                连接断开
   ├─ 校验群成员           ├─ JSON 反序列化          ├─ 移除 session
   ├─ session 入集合        ├─ DFA 敏感词过滤          ├─ 广播离开系统消息
   └─ 广播加入系统消息       ├─ CompletableFuture      └─ 空集合清理
                            │   .runAsync(入库)
                            └─ 广播消息至同群
                               （排除发送方）
```

### 技术细节

- **Session 管理**：`ConcurrentHashMap<Long, Set<WebSocketSession>>`，`computeIfAbsent` + `ConcurrentHashMap.newKeySet()` 保证线程安全
- **异步入库**：`CompletableFuture.runAsync()` 异步持久化消息，不阻塞广播（消息 ID 在广播时可能为 null）
- **广播策略**：消息广播排除发送方本人，系统消息广播给全群（包括触发者）
- **敏感词过滤**：WebSocket 消息同样经 `SensitiveWordUtil.filter()` 处理后入库和广播
- **JSON 构造**：手动字符串拼接（非 ObjectMapper），避免依赖循环和序列化开销
- **安全校验**：握手拦截器校验 token + 群成员身份，双重保障

**目标**：为后台管理系统提供数据看板接口，展示平台核心运营指标。

```
返回:
{
  "totalUsers": 1280,          // 平台总用户数
  "todayPosts": 56,             // 今日新增帖子数
  "topActiveUsers": [...]       // Top 5 活跃用户
}
```

### 查询策略

| 指标 | 实现方式 | 说明 |
| --- | --- | --- |
| 总用户数 | LambdaQuery `userMapper.selectCount()` | 单表 COUNT，`@TableLogic` 自动过滤已删除 |
| 今日帖子数 | LambdaQuery `postMapper.selectCount()` | `between(LocalDateTime, todayStart, todayEnd)` |
| Top 5 活跃用户 | XML 手写 SQL | UNION ALL 跨 `forum_post` + `forum_comment` 聚合 |

### 为什么 Top 5 必须手写 SQL

LambdaQuery 不支持 UNION ALL 和跨表聚合。如果拆成两个查询，在 Java 层合并排序，当数据量大时需要拉取全表数据，不可取。手写 SQL 在数据库层完成聚合，一次查询返回 5 行结果。

```sql
SELECT u.id, u.username, u.avatar_url, SUM(sub.cnt) AS total
FROM sys_user u
INNER JOIN (
    SELECT user_id, COUNT(*) AS cnt FROM forum_post WHERE is_deleted = 0 GROUP BY user_id
    UNION ALL
    SELECT user_id, COUNT(*) AS cnt FROM forum_comment WHERE is_deleted = 0 GROUP BY user_id
) sub ON u.id = sub.user_id
WHERE u.is_deleted = 0
GROUP BY u.id, u.username, u.avatar_url
ORDER BY total DESC
LIMIT #{limit}
```

## 八、热门推荐（HN 热度算法）

**目标**：帖子列表不再仅按时间倒序，而是基于 Hacker News 热度衰减算法推荐高质量内容。

### 算法公式

```
Score = (likeCount − 1) / (T + 2)^1.8

T = 发布时间到当前的小时数
G = 1.8（重力常数）
```

- 0 赞：负分 → 永久沉底
- 1 赞：0 分
- ≥ 2 赞：正分，随时间衰减。新帖有天然优势，老帖需要持续点赞维持排名

### 变更清单

| 文件 | 动作 | 说明 |
| --- | --- | --- |
| `config/SchedulingConfig.java` | **新建** | `@EnableScheduling` 启用定时任务 |
| `module/post/task/HotPostScheduler.java` | **新建** | `@Scheduled(fixedRate = 600000)` 每 10 分钟计算热度，写入 Redis ZSet |
| `module/post/service/PostService.java` | 修改 | 新增 `getHotPosts(int pageNum, int pageSize)` 接口 |
| `module/post/service/impl/PostServiceImpl.java` | 修改 | 注入 `RedisTemplate`，实现 Redis ZSet 分页 + DB 批量查询 + VO 组装 |
| `module/post/controller/PostController.java` | 修改 | 新增 `GET /api/posts/hot` 端点 |

### 架构设计

```
                         HotPostScheduler
                    (@Scheduled, 每 10 分钟)
                              │
          ┌───────────────────┤
          ▼                   ▼
    查询 DB（近 7 天）    计算 HN 热度分
          │                   │
          └───────────────────┘
                    │
            Redis ZSet (hot:posts)
          member=postId, score=热度分
                    │
          ┌─────────┴─────────┐
          ▼                   ▼
   GET /api/posts/hot    ZREVRANGE 分页
          │                   │
    ZCARD 拿总数         批量 selectBatchIds
          │                   │
          └───────────────────┘
                    │
              组装 PageInfo<PostDetailVO>
```

### 关键实现细节

**定时任务流程**：
```
1. SELECT id, like_count, created_at FROM forum_post
   WHERE is_deleted = 0 AND created_at > NOW() - 7 DAY
2. 遍历计算: score = (likeCount - 1) / Math.pow(t + 2, 1.8)
3. redisTemplate.delete("hot:posts")
4. redisTemplate.opsForZSet().add("hot:posts", postId, score)
5. log 完成
```

**接口分页流程**：
```
1. ZCARD → total
2. ZREVRANGE hot:posts start end WITHSCORES → postIds（按热度降序）
3. postMapper.selectBatchIds(postIds) → DB 批量查
4. 按 Redis 顺序重排 → 组装 VO（作者信息 + 点赞状态）
5. 手动构造 PageInfo（不使用 PageHelper，因为数据源是 Redis）
```

**容错设计**：
- 定时任务 try-catch：Redis 挂了不影响主流程，log.error 记录
- 接口 try-catch：Redis 不可用时返回空 PageInfo（不报 500）
- 已删除帖子过滤：`selectBatchIds` 后过滤 `is_deleted = 1`
- 冷启动：首次部署 Redis 为空，返回空列表

## 九、文件上传

**目标**：为帖子的富文本内容提供图片和视频上传能力，与 XSS 清洗管道配合实现图文混排。

### 变更清单

| 文件 | 动作 | 说明 |
| --- | --- | --- |
| `application.properties` | 修改 | 新增 `spring.servlet.multipart.*` + `app.upload.*` 配置 |
| `config/WebResourceConfig.java` | **新建** | 映射 `/uploads/**` 到磁盘路径，响应静态文件访问 |
| `module/upload/controller/UploadController.java` | **新建** | 图片/视频上传接口，三重文件校验 |

### 接口设计

```
POST /api/upload/image   →  图片上传，≤10MB，支持 jpg/jpeg/png/gif/webp
POST /api/upload/video   →  视频上传，≤100MB，支持 mp4/webm

返回: { "code": 200, "data": { "url": "/uploads/images/202605/abc.png" } }
```

### 安全校验（三重防线）

```
1. 扩展名白名单 → 仅允许指定格式
2. MIME 类型校验 → 检测 Content-Type
3. 文件头魔数校验 → 读取前 4-12 字节验证真实类型（防止伪造扩展名）
```

| 格式 | 魔数（Hex） |
| --- | --- |
| JPEG | `FF D8 FF` |
| PNG | `89 50 4E 47` |
| GIF | `47 49 46` |
| WebP | `52 49 46 46 ... 57 45 42 50` |

### 存储规则

- 文件重命名为 `UUID.{ext}`，防止路径穿越
- 按年月分目录：`uploads/{images,videos}/{yyyyMM}/`
- 文件通过 `/uploads/**` 直接访问（Spring ResourceHandler 映射）

### 前后端对接流程

```
Markdown 编辑器（前端）
  → 选择/粘贴图片
  → POST /api/upload/image (FormData)
  → 拿到 url
  → 插入 ![](/uploads/images/202605/xxx.png)
  → 编辑器渲染为 <img src="/uploads/images/202605/xxx.png">
  → 帖子提交 content = HTML
  → Jackson Jsoup 清洗（移除 onerror 等）
  → 存入 MySQL
  → 前端查看帖子: v-html / dangerouslySetInnerHTML
```

> 无需新增依赖，Spring Boot 内置的 multipart + NIO 文件写入已足够。

## 十、完整请求处理管道

```
HTTP 请求
  │
  ├─ 1. CorsFilter          (跨域放行)
  ├─ 2. SaTokenInterceptor  (JWT 鉴权，白名单放行)
  ├─ 3. @RateLimit AOP      (Redis ZSET 滑动窗口限流)
  │
  ▼
  ├─ 4. Jackson 反序列化
  │     └─ XssStringDeserializer → Jsoup.clean(Safelist.relaxed())
  │     └─ Long → String (前端序列化)
  ├─ 5. @Valid / @Validated (JSR-303 参数校验)
  │
  ▼
  ├─ 6. Controller → Service
  │     └─ SensitiveWordUtil.filter() → DFA 敏感词替换
  │     └─ BusinessException → GlobalExceptionHandler → Result JSON
  │
  ▼
  └─ 7. MyBatis-Plus → MySQL
        └─ MybatisPlusMetaObjectHandler (createdAt/updatedAt 自动填充)

WebSocket 请求
  │
  ├─ 1. HandshakeInterceptor (Satoken 校验 + 群成员校验)
  │
  ▼
  ├─ 2. afterConnectionEstablished (广播系统消息)
  ├─ 3. handleTextMessage → JSON 解析 → DFA 敏感词过滤 → 异步入库 → 广播
  ├─ 4. afterConnectionClosed (广播系统消息)
```
